# Render Deployment

CommerceLens includes a Render Blueprint in `render.yaml` for a production-style
hosted deployment:

- `commercelens-api`: FastAPI web service
- `commercelens-worker`: background monitoring worker
- `commercelens-db`: managed Postgres database

## Deploy

1. Push `main` to GitHub.
2. In Render, create a new Blueprint from this repository.
3. Use the root `render.yaml` file.
4. Review the generated services and database.
5. Replace `COMMERCELENS_USER_AGENT` with a real operational contact email.
6. Apply the Blueprint.

Render generates `COMMERCELENS_ADMIN_TOKEN` automatically for the API service.
Save that value from the service environment after the first sync. You need it
to create accounts, projects, members, API keys, and to open the dashboard.

## Services

The API service runs:

```bash
uvicorn commercelens.api.main:app --host 0.0.0.0 --port "$PORT"
```

The worker service runs:

```bash
commercelens worker --poll-seconds 60 --limit 25
```

Both services use the same managed Postgres connection through
`COMMERCELENS_DATABASE_URL`.

## Required Production Values

The Blueprint sets these values:

```text
COMMERCELENS_ENV=production
COMMERCELENS_STORE_BACKEND=postgres
COMMERCELENS_REQUIRE_API_KEY=true
COMMERCELENS_DATABASE_URL=<Render private Postgres connection string>
DATABASE_URL=<Render private Postgres connection string>
COMMERCELENS_POSTGRES_DSN=<Render private Postgres connection string>
COMMERCELENS_ADMIN_TOKEN=<generated by Render>
COMMERCELENS_DEFAULT_TIMEOUT_SECONDS=20
```

Update this value before real customer traffic:

```text
COMMERCELENS_USER_AGENT=CommerceLens/0.9 (+mailto:ops@yourdomain.com)
```

## Smoke Test

Replace `https://YOUR-API.onrender.com` and `$ADMIN_TOKEN`.

Run the automated smoke test:

```bash
python scripts/smoke_deploy.py \
  --base-url https://YOUR-API.onrender.com \
  --admin-token "$ADMIN_TOKEN" \
  --owner ops@example.com
```

The script checks `/health`, `/ready`, account creation, project creation, API
key creation, product extraction, usage summary, and dashboard access.

Manual checks:

```bash
curl https://YOUR-API.onrender.com/health
curl https://YOUR-API.onrender.com/ready
```

Create an account:

```bash
curl -X POST https://YOUR-API.onrender.com/v1/accounts \
  -H "Content-Type: application/json" \
  -H "X-Admin-Token: $ADMIN_TOKEN" \
  -d '{"name":"Demo Account","owner":"ops@example.com"}'
```

Create an API key:

```bash
curl -X POST https://YOUR-API.onrender.com/v1/api-keys \
  -H "Content-Type: application/json" \
  -H "X-Admin-Token: $ADMIN_TOKEN" \
  -d '{"name":"demo key","account_id":"acct_REPLACE","scopes":["*"]}'
```

Run a product extraction:

```bash
curl -X POST https://YOUR-API.onrender.com/v1/extract/product \
  -H "Content-Type: application/json" \
  -H "X-API-Key: cl_REPLACE" \
  -d '{"url":"https://example.com/products/demo","html":"<html><script type=\"application/ld+json\">{\"@context\":\"https://schema.org\",\"@type\":\"Product\",\"name\":\"Demo Product\",\"offers\":{\"@type\":\"Offer\",\"price\":\"19.99\",\"priceCurrency\":\"USD\"}}</script></html>"}'
```

Open the private dashboard:

```text
https://YOUR-API.onrender.com/dashboard?admin_token=ADMIN_TOKEN
```

## Operational Notes

- The Blueprint uses paid instance classes because background workers and managed
  Postgres are production resources.
- Keep the database private with `ipAllowList: []`.
- Use the web service's `/ready` path as the health check.
- The services use `autoDeployTrigger: checksPass`, so Render waits for GitHub
  checks to pass before automatic deploys.
- Put the dashboard behind SSO, VPN, or a trusted edge before customer use.
- Start with one worker. Add more only after adding distributed job claiming.
